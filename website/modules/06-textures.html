📚 Core Concepts

The Camera is Your Viewpoint

In Three.js, the camera determines what the user sees. Unlike real cameras that capture light, Three.js cameras define a viewing frustum — a pyramid-shaped volume of space that gets rendered. Objects inside this frustum are visible; objects outside are "culled" (not rendered) for performance.

Think of it like looking through a window. The window frame determines what you can see, and you can only see things between the glass and the horizon — not things behind you or past the horizon. The camera's parameters control the shape and size of that window.

The most common camera is PerspectiveCamera, which mimics how human eyes see the world: objects farther away appear smaller. The alternative is OrthographicCamera, which has no perspective distortion — like architectural blueprints where parallel lines stay parallel.

PerspectiveCamera Parameters

📷
The Four Camera Parameters

new THREE.PerspectiveCamera(fov, aspect, near, far)

  • FOV — Vertical viewing angle in degrees (75° is a good default)
  • Aspect — Width / Height of your viewport (match the window)
  • Near — Minimum render distance (0.1 typical, never 0)
  • Far — Maximum render distance (1000 typical)

FOV (Field of View) is the vertical viewing angle. Higher values give a wider fish-eye view, lower values give a narrow telephoto zoom. Human eyes see roughly 50-60°. Game cameras use 75-90° for immersion.

Aspect Ratio should always match your viewport: window.innerWidth / window.innerHeight. Get this wrong and everything stretches.

Near/Far Clipping Planes define the render range. Setting near to 0 causes Z-fighting (depth buffer precision issues). Setting far too high wastes GPU resources on invisible objects.

OrbitControls: Interactive Navigation

Portfolio sites need to feel alive. OrbitControls wires up all the standard 3D interactions: left-click to rotate, right-click to pan, scroll to zoom, and touch gestures for mobile.

The camera orbits around a target point (default: origin). The most important setting is damping — it adds inertia so the camera keeps gliding after you stop dragging. Without damping, movement feels abrupt. With it, your scene feels polished and premium.

Key configuration: minDistance/maxDistance for zoom limits, maxPolarAngle to prevent going underground, autoRotate for showcase hero sections, and enablePan to restrict to orbit-only.

Camera Animation

Cameras are just another Object3D — they have position, rotation, and can be animated like any object. Use linear interpolation (lerp) combined with easing functions for smooth, natural-feeling transitions between camera positions.

Key Insight

The camera is just another Object3D — it has position, rotation, and can be parented to groups. You can animate it like any other object in your scene.

🔑 Key Terminology

FOV
Field of View — the vertical viewing angle in degrees. 75° is normal, 120° is wide-angle, 30° is telephoto zoom.
Frustum
The pyramid-shaped volume between near and far clipping planes. Only objects inside this volume get rendered.
Clipping Plane
Near/far boundaries that control render range. Objects outside these planes are invisible. Near should never be 0.
OrbitControls
Mouse/touch controls to rotate the camera around a target point. The standard interaction model for 3D viewers and portfolios.
Damping
Inertia effect that makes the camera glide smoothly after input stops. Requires controls.update() in the animation loop.
Lerp
Linear interpolation — smoothly blending between two values. Used to animate camera from position A to position B.

💻 Code Examples

camera-setup.js Working Example 
import * as THREE from 'three';

const scene = new THREE.Scene();

// Create camera — these are the most common "good defaults"
const camera = new THREE.PerspectiveCamera(
    75,                                      // FOV: wider than human eye
    window.innerWidth / window.innerHeight,  // Aspect: match browser window
    0.1,                                     // Near: very close to camera
    1000                                     // Far: render up to 1000 units
);

// Position camera — default (0,0,0) puts you INSIDE objects!
camera.position.set(5, 5, 5);
camera.lookAt(0, 0, 0);  // Point at scene origin

// Dynamically change FOV (zoom effect)
camera.fov = 50;  // Narrower = more zoomed in
camera.updateProjectionMatrix();  // REQUIRED after changes!

const renderer = new THREE.WebGLRenderer();
renderer.setSize(window.innerWidth, window.innerHeight);
document.body.appendChild(renderer.domElement);

const cube = new THREE.Mesh(
    new THREE.BoxGeometry(1, 1, 1),
    new THREE.MeshBasicMaterial({ color: 0x00ff00, wireframe: true })
);
scene.add(cube);

function animate() {
    requestAnimationFrame(animate);
    cube.rotation.y += 0.01;
    renderer.render(scene, camera);
}
animate();
What's happening: We create a camera with typical settings and pull it back from the origin. The critical detail: whenever you change camera properties like fov at runtime, you must call updateProjectionMatrix() — otherwise the change is silently ignored.

orbit-controls.js Working Example 
import { OrbitControls } from 'three/examples/jsm/controls/OrbitControls.js';

// After creating camera and renderer...
const controls = new OrbitControls(camera, renderer.domElement);

// Enable damping for smooth, premium feel
controls.enableDamping = true;
controls.dampingFactor = 0.05;  // Lower = more glide

// Prevent extreme zoom
controls.minDistance = 2;
controls.maxDistance = 20;

// Stop camera from going below the floor
controls.maxPolarAngle = Math.PI / 2;

// Disable panning — orbit only (cleaner for portfolios)
controls.enablePan = false;

// Auto-rotate when idle
controls.autoRotate = true;
controls.autoRotateSpeed = 2;

// CRITICAL: update every frame when damping is on
function animate() {
    requestAnimationFrame(animate);
    controls.update();  // Calculates inertia each frame
    renderer.render(scene, camera);
}
animate();
What's happening: OrbitControls listens for mouse/touch events on the renderer's canvas. Damping creates smooth gliding by gradually reducing velocity each frame — but only if you call controls.update() in the animation loop.
camera-animation.js Working Example 
// Smooth camera movement with ease-in-out
function moveCameraTo(targetPos, duration = 1000) {
    const startPos = camera.position.clone();
    const startTime = Date.now();

    function animateCamera() {
        const elapsed = Date.now() - startTime;
        const progress = Math.min(elapsed / duration, 1);

        // Ease-in-out: accelerate then decelerate
        const eased = progress < 0.5
            ? 2 * progress * progress
            : 1 - Math.pow(-2 * progress + 2, 2) / 2;

        // Blend between start and target
        camera.position.lerpVectors(startPos, targetPos, eased);

        if (progress < 1) {
            requestAnimationFrame(animateCamera);
        }
    }
    animateCamera();
}

// Move camera over 2 seconds
moveCameraTo(new THREE.Vector3(10, 5, 10), 2000);
What's happening: We capture the starting position, then each frame calculate progress (0→1). The easing function transforms linear progress into smooth acceleration/deceleration. lerpVectors blends between start and end positions.
💡 Try it: Set up OrbitControls with and without damping. Feel the difference — damping makes your scene feel professional instead of mechanical.

Exercise 1: Camera Parameter Exploration

⏱️ ~15 min

Goal: Understand how FOV and clipping planes affect the view

  1. Create a scene with 5 cubes at different distances (z = -5, 0, 5, 10, 20)
  2. Give each cube a different color so you can tell them apart
  3. Start with FOV = 75, near = 0.1, far = 100
  4. Change FOV to 30 — observe the "zoomed in" telephoto effect
  5. Change FOV to 120 — observe the wide-angle distortion
  6. Set far = 15 — watch distant cubes vanish
  7. Set near = 10 — watch close cubes disappear

Exercise 2: OrbitControls Configuration

⏱️ ~25 min

Goal: Configure OrbitControls for a portfolio-like experience

  1. Create a scene with a sphere or torus at the center
  2. Add OrbitControls with damping (dampingFactor = 0.05)
  3. Set min/max distance to prevent extreme zoom
  4. Set maxPolarAngle = Math.PI / 2 to prevent underground view
  5. Disable panning and enable auto-rotation

controls.update() must be in your animation loop — auto-rotation and damping both require it.

🤔 Reflection Questions

Q1: What happens if you set the near clipping plane to 0?

✅ Correct! Setting near to 0 causes the depth buffer to lose precision, resulting in Z-fighting where overlapping surfaces flicker.

❌ Setting near to 0 causes depth buffer precision issues. The near/far ratio determines depth precision — a near plane at 0 means infinite ratio, causing Z-fighting.

Q2: When would you use OrthographicCamera instead of PerspectiveCamera?

✅ Correct! OrthographicCamera preserves parallel lines — essential for architectural plans, isometric games, and technical visualization.

❌ OrthographicCamera is used when you need parallel lines to stay parallel — like architectural blueprints, isometric games, or CAD software.

Q3: Why must you call controls.update() every frame when damping is enabled?

✅ Correct! Damping works by reducing velocity each frame — update() is where that calculation happens. Without it, damping (and auto-rotate) won't work.

❌ Damping needs update() because that's where the inertia calculation happens each frame. The controls gradually reduce velocity — that math runs inside update().
